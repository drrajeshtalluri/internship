The AI Gauntlet: Advanced Challenges

AI can build anything you ask. But can it build *exactly* what a client wants? Let's find out.

From Builder to Architect

You've seen that AI is an amazing tool for writing code quickly. That's true! But a professional developer's most important job isn't just writing code—it's designing systems, solving complex problems, and debugging when things go wrong.

An AI is like a brilliant but very literal assistant. It can build a wall if you ask it to, but it can't design the whole house. That's the architect's job. Your job.

These next two challenges are your architect's blueprints. They have very specific, interconnected rules that a simple prompt like "make a number guessing game" will fail to capture. Your mission is to use any tool you want—including AI—to build these exact specifications. The real test is not in getting the final product, but in figuring out how to direct, correct, and combine the code to meet these complex demands.

Challenge 1: The "Guess Master 3000"

This isn't a simple guessing game. It's a dynamic application with multiple states and a detailed user interface. You must build it to these exact specifications.

Core Logic Specifications

  • Difficulty Levels: At the start, the player must be presented with three buttons: "Easy (1-50, 10 guesses)", "Medium (1-100, 7 guesses)", and "Hard (1-200, 8 guesses)". Clicking one starts the game with the selected settings.
  • Temperature Feedback: Instead of "Too high/low," the feedback must be based on proximity:
    • Hot! (within 5 of the secret number)
    • Warm (within 15 of the secret number)
    • Cold (within 30 of the secret number)
    • Freezing! (more than 30 away)
  • Hint System: A "Get a Hint" button must appear only after the 3rd wrong guess. Clicking it costs the player 2 guesses and reveals if the secret number is even or odd. The button should disappear after being used once.

User Interface (UI) Specifications

  • Guess History: All previous guesses must be displayed in a list on the page. Each guess in the list must show the number guessed and the "temperature" feedback it received (e.g., "75 - Cold").
  • Dynamic Guesses Remaining Bar: You must create a visual progress bar that shows how many guesses are left. This bar should start full (green), turn yellow when 3 or fewer guesses remain, and turn red on the last guess.
  • Input Validation: The guess input box must prevent the user from guessing the same number twice. If they try, an error message "You already guessed that number!" must appear below the input box for a few seconds.
  • Win/Loss Screen: The win or loss message must not just appear as text. It must be a modal pop-up that appears over the game. This modal must show the final outcome and contain a "Play Again" button.
  • Full Reset: The "Play Again" button in the modal must completely reset the game to the initial difficulty selection screen *without reloading the page*. This means all variables, the guess history, and the progress bar must be reset to their starting states using JavaScript.

Challenge 2: The Chronos Key

This is not a simple, linear story. It's a small game system with state that changes based on player actions. The UI must be dynamic and reflect the game's state at all times.

Core Logic Specifications

  • Inventory System: The player must be able to collect items by making certain choices. The game must track these items in an inventory.
  • Conditional Choices: Some choices must only be available if the player has a specific item in their inventory. For example, a "Unlock the chest" choice should only appear if the player has a "key".
  • Stateful Environment: An action in one room must be able to change the state of another room. For example, pulling a lever in the "Machine Room" might change the description of the "Main Hall" to say "a secret panel has opened".
  • Time Limit: The player only has 15 moves to finish the game. Each action (moving to a new room, taking an item) costs 1 move. If the counter reaches 0, the game is over.

User Interface (UI) Specifications

  • Three-Panel Layout: The UI must be divided into three distinct, always-visible areas:
    1. Main Story Panel: A large area where the room/scene descriptions are displayed.
    2. Choices Panel: An area below the story where choices are displayed as clickable buttons, not text prompts.
    3. Status Sidebar: A sidebar on the right that displays:
      • Inventory: A list of items the player is carrying. This list must update immediately when an item is picked up.
      • Moves Remaining: A counter that ticks down with each action.
  • Dynamic Buttons: The choice buttons must not be static. Your JavaScript must create, show, or hide buttons based on the current room and the player's inventory. If a choice becomes unavailable, its button should be removed from the Choices Panel.
  • Visual Transitions: When the player moves to a new room, the text in the Main Story Panel must smoothly fade out, and the new room's text must smoothly fade in. (This requires simple CSS transitions triggered by JavaScript).

The Final Requirement: Justify Your Code

This is the most important rule. For either challenge you choose, you must comment your JavaScript code. Don't just write what a line does. Write WHY you are doing it that way. This proves you are the architect.

// GOOD COMMENT: Explains the "why" // I'm using a function here to reset the game state. This is better than // resetting the variables one by one because I can just call resetGame() // from multiple places (like the 'Play Again' button and at the start). function resetGame() { ... } // BAD COMMENT: Only says what the code already shows // This function resets the game. function resetGame() { ... }

Being able to explain your decisions is the difference between writing code and being a developer.
